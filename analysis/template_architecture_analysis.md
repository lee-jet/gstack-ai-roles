# gstack 专家提示词模板架构分析报告

本报告对提取的 13 个 gstack 专家角色提示词进行了深度对比，旨在揭示其背后的工程哲学、通用模式以及针对不同角色的差异化设计。

---

## 1. 核心共性 (Core Commonalities)

尽管每个角色的职责不同，但所有模板都遵循一套统一的“元架构”：

### A. 使命驱动型开端 (Mission-Driven)
每个文件都以 `# Role: [角色名]` 开头，并紧跟一个 **Mission (使命)** 声明。这为 AI 设定了基本的身份认知和目标优先级（例如：CEO 优先考虑产品价值，EM 优先考虑技术严谨性）。

### B. “煮湖论” (Boil the Lake) 哲学
大多数模板显式或隐式地包含了“确保回复的完整性”要求。这是 gstack 的核心精神：不放过任何细节，对问题进行地毯式搜索和深度挖掘。

### C. 结构化阶段执行 (Phased Execution)
所有核心角色都不使用单次 Prompt，而是采用 **Phase 1 -> Phase 2 -> Phase 3** 的线性或循环工作流。
- **阶段 1：上下位收集/审计** (Orientation/Audit)
- **阶段 2：执行/分析** (Execution/Analysis)
- **阶段 3：验证/总结** (Verification/Summary)

### D. 硬性准则约束 (Important Rules)
每个模板末尾都有一个 `Important Rules` 章节，用于设定技术红线（如：严禁 Force Push、零容忍 Secret）和行为准则（如：反阿谀奉承）。

---

## 2. 差异化设计 (Differentiation)

差异化主要体现在**认知模式**、**工具侧重**以及**产出物风格**上：

### A. 认知模式对比
| 角色类型 | 核心关注点 | 典型思维模式 |
| :--- | :--- | :--- |
| **决策层 (CEO/PM)** | 产品价值、范围、用户体验 | “十星级体验”、“核能范围挑战” |
| **管理层 (EM/Architect)** | 架构一致性、技术债、测试覆盖 | “先分析再修复”、“模块化/原子化” |
| **审计层 (CSO/QA/Reviewer)** | 风险、Bug、安全性、边缘情况 | “寻找破绽”、“回归测试驱动” |
| **创意层 (Designer)** | 视觉灵魂、系统连贯性、创意风险 | “审美标定”、“像素级校准” |

### B. 差异化工具配置
- **视觉类角色** (QA/Designer)：深度调用 `Chromium/Browse` 进行真实渲染对比。
- **逻辑类角色** (EM/Reviewer/CSO)：深度依赖 `Grep`、`Git log` 和 `Diff` 分析。
- **分析类角色** (Retro/Performance)：专注于 `Git` 数据挖掘和指标计算。

### C. 产出物差异化
- **CEO**: 产出建议书、远景描述、对实施计划的“扩张/削减”决策。
- **EM/Reviewer**: 产出审计报告、改进建议及具体的代码修正。
- **Designer/QA**: 产出截图对比、视觉评分表、UI 规范 (`DESIGN.md`)。

---

## 3. 架构总结

gstack 的提示词模板架构实际上是一套**“软件工程经验的封装协议”**。它通过：
1. **身份锚定**（角色定义）
2. **流程约束**（操作阶段）
3. **工具赋能**（特定 CLI 调用）
4. **文化根植**（Boil the Lake 等哲学）

将原本模糊的“氛围驱动 (Vibe Coding)”转化为了**高度可重复、可审计的工业化开发流程**。
