# 🌌 gstack-ai-roles: 精英工程引擎 (Elite Engineering Engine)

本引擎实现了 **“AI 软件工厂”** 范式。它通过一系列专业子代理的严谨交接，以及“煮湖论”式的工程标准，将高层愿景转化为稳健、可交付的生产代码。

## 🏗️ 运营单元 (Operational Units)

| 阶段 | 专业领域 | 核心产出 (Artifact) | gstack 对齐 |
| :--- | :--- | :--- | :--- |
| **1. 思考 (Think)** | 战略产品对齐 | `DESIGN.md` | CEO / Office Hours |
| **2. 规划 (Plan)** | 架构蓝图设计 | `PLAN.md` | 工程经理 / 设计审计 |
| **3. 编码 (Build)** | 精准实施 | `代码库` | 开发者 / 视觉实现 |
| **4. 评审 (Review)** | 对抗式审计 (CSO) | `审计报告` | 资深架构师 / 安全官 |
| **5. 测试 (Test)** | 功能压力验证 (QA) | `测试通过` | QA 组长 / 浏览器专家 |
| **6. 发布 (Ship)** | 交付治理 (Retro) | `部署上线` | 发布工程师 / 文档工程师 |

## 🚀 执行模式

- **全量冲刺**：“工程化实现 [功能描述]” — 自动运行全生命周期。
- **部分冲刺**：“跳过思考阶段，从规划开始” — 基于现有文档恢复。
- **单单元操作**：`/think`, `/plan`, `/review`, `/ship`。

## 💎 工程护栏 (Guardrails)
- **契约先行 (Contract-First)**：在 `PLAN.md` 中的系统拓扑与状态机锁定前，严禁任何代码分发。
- **对抗审计**：每一项变更都必须通过“Codex 式”的多模型对抗挑战。
- **煮湖论 (Boil the Lake)**：追求 100% 的鲁棒性。处理每一个边界情况。机器不会疲倦，不要满足于“还行”。

请查阅 `references/prompts.md` 获取每个运营单元的内部作业规范 (SOP)。
